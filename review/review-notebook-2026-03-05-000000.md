## Summary

This is a messy but genuinely thoughtful exploratory notebook. The ML intuition is mostly sound — the progression from Linear Regression → Random Forest → XGBoost with log transform is well-motivated, and the decision to introduce `state_30d_baseline` as an anchor shows real domain insight. But the notebook has serious reproducibility and correctness problems: confirmed data leakage in a key cell, `TimeSeriesSplit` applied incorrectly to panel data, a model that was saved before its training was finished, critical configuration not exported to metadata, and variables silently reused across dozens of cells that make the reported metrics untrustworthy. If someone ran this notebook top-to-bottom today, they would not reproduce the numbers claimed in the README.

## Findings

### 1. LOGIC — Confirmed Data Leakage (Cell 173)

```python
y_log = np.log1p(y)
rf1.fit(X, y_log)          # ← fits on ALL data
y_pred = np.expm1(rf1.predict(X_test))
```

`rf1` is fitted on the entire `X` (including test rows), then evaluated on `X_test`. The model has already seen the test labels during training. This is textbook data leakage. The metrics from this cell are inflated and meaningless. The pattern appears again subtly later when rolling features are computed on the full dataset before splitting.

---

### 2. LOGIC — `TimeSeriesSplit` Is Wrong for Panel (Multi-State) Data

**Cells**: 147, 244, 264, 283

The dataset is **multi-state panel data** — rows from different states for the same date are interleaved. `TimeSeriesSplit` splits by row index, not by calendar date. Since the data is sorted by `["states", "date"]` (cell 249), the split puts early states (alphabetically first, e.g. Andhra Pradesh) in training and later states in test — **not** earlier dates in training and later dates in test. Rows from March 2020 for one state will appear in training while rows from March 2020 for another state appear in test. This is not a temporal split; it's a geographical split masquerading as one, and it causes optimistic metrics because the model sees "future" data from one state while being tested on "past" data from another.

The correct approach: split by **date** (e.g., all rows with `date < cutoff` → train), not by row index.

### 7. ASSUMPTIONS — Rolling Features Computed on Full Dataset Before Train/Test Split

**Cells 51–52 (energymet_7d_avg, generation_7d_avg) and 249–252:**
```python
merged["energymet_7d_avg"] = merged.groupby("states")["energy met (mu)"].transform(
    lambda x: x.rolling(window=7, min_periods=1).mean()
)
```

`transform` with `rolling` here uses `.mean()` without `.shift(1)`. This means the 7-day average **includes the current day**. For training data where the target is `energy met (mu)`, a 7-day rolling mean that includes the current day's value is partial target leakage. The correct approach is `x.shift(1).rolling(7)` to ensure only past data is used — which is actually done correctly in the later rolling features (cells 250–251), but not here.

### 13. EDGE CASE — `CDD` Computed in Celsius Despite Temperature Stored in Kelvin

**Cell 243:**
```python
df_total["CDD"] = (df_total["2m_temperature_mean"] - 24).clip(lower=0)
```

Temperature is in Kelvin (~300K for 27°C). Subtracting 24 from ~300 gives a CDD of ~276, which is nonsensical. The threshold should be `297.15` (24°C in Kelvin). The `app.py` code correctly uses `297.15` at inference time (`app.py:869`) — but the training data uses `24`. This is a **training-inference skew**: the model was trained with a wrong (but consistent) CDD scale, and the app uses a different (more correct) scale. The model learned on CDD values of ~276, but inference feeds CDD values of ~0–5.

### 14. EDGE CASE — `state_30d_baseline` Is Added After `xgb1` Is Saved

The model `xgb2` is trained with `state_30d_baseline` as a feature. But `model_metadata.json` (cells 290–291) is generated from `xgb2`'s metrics but **does not record the feature list**. The metadata only stores aggregate RMSE, not which features the model expects. This means there is no programmatic way to verify feature compatibility from the metadata alone.

## Positive Notes

1. **TimeSeriesSplit is used for CV**, not random shuffle split — this shows awareness that temporal ordering matters and prevents future-leakage in the cross-validation loop itself (even if the panel-data split is wrong).

2. **Log-transform motivation is clearly explained** (cells 172, 183): the reasoning from heteroscedasticity to `log1p` is well-documented and correct. This is a textbook-quality explanation.

3. **The `state_30d_baseline` idea is genuinely clever** (cell 277): using a lagged 30-day rolling mean of the state's own demand as a baseline anchor solves the cross-state scale difference problem without needing separate models per state. This is domain-driven feature engineering done right.

4. **Residual analysis is thorough**: residual vs. fitted, QQ plots, and time-series overlays are all present and discussed. The interpretation of "heavy tails = unpredictable events" (cells 238, 179) is correct.

5. **Holiday and extreme heat flags are contextually motivated**: the explicit curated holiday list (cell 196) and the 95th-percentile threshold for extreme heat (cell 198) show real domain thinking rather than naive feature engineering.

6. **Multi-source comparison is visualized** (cell 131): the source breakdown (Hydro, Solar, Coal, etc.) with rolling averages is a nice EDA touch that most notebooks skip.

7. **The demand-vs-generation correlation correction** (cells 64–68) is a solid piece of exploratory reasoning: identifying that the weak state-level correlation is a disaggregation artifact, then confirming near-perfect national correlation, is good statistical thinking.

---

## Verdict

**This notebook reads like a real investigation, not a polished tutorial.** The thinking is genuine. The mistakes — data leakage, wrong panel split, reused variable names — are the normal mistakes of someone learning by doing and not stopping to refactor. The two issues that actually matter for the deployed model are:

1. **CDD training/inference skew** (item 13): the model was trained on CDD = `temp_K - 24`, the app uses CDD = `temp_K - 297.15`. These are different scales. This likely has significant impact on predictions.
2. **`extreme_heat` threshold not persisted to metadata** (item 5): re-computed per forecast window instead of using the training threshold.

Everything else is process debt — solvable by converting the final pipeline (data prep → feature engineering → train → evaluate → save) into a clean, standalone script separate from the exploration. The notebook should stay as the EDA artifact; the production pipeline should not live inside it.
