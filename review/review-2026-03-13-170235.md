## Summary

The pre-modelling pipeline is well-structured and shows genuine domain understanding of India's electricity grid. The Zenodo dataset integration is thoughtful, the EDA uncovers and corrects a real analytical mistake (state-level vs. national correlation), and the feature engineering is grounded in physical reasoning (CDD, holidays, heat waves). However, there are several reproducibility, correctness, and maintainability issues — most notably a silent data leakage risk in how the post-merge dataframe is filtered and reused, inconsistent handling of the `source` column, and a weighted-average implementation that is never actually used in the final model.

**Overall rating: 7 / 10** — Solid exploratory work with targeted fixes needed before hardening into a production pipeline.

## Findings

### 1. Logic

#### [HIGH] `df_total` is re-derived inconsistently throughout the notebook
**Location:** Cells 17, 18, 20, 22, 27, 36, 39, 85–86
Multiple cells independently re-filter `merged` with `merged['source'] == 'Total'` and assign it to `merged_total` or `df_total`. These are not the same object — some are copies, some are views, some include weather columns (post-Zenodo merge), some do not. This creates a false impression that all analyses operate on the same filtered dataset.

```python
# Cell 17
merged_total = merged[merged['source'] == 'Total']   # pre-Zenodo merge

# Cell 85
df_total = df[df['source']=='Total']                 # post-Zenodo merge
```

**Risk:** EDA findings (e.g., correlations of 0.472 vs. 0.60) are silently computed on different slices with different column sets, making the results non-comparable and potentially misleading.

**Fix:** Define a single `df_total` at one canonical point (post-Zenodo merge) and use it consistently throughout. Remove redundant re-derivations.

#### [HIGH] Population-weighted averages computed but never used
**Location:** Cells 74–82
A well-implemented demand-weighted national average is computed for temperature, solar radiation, cloud cover, and UTCI:

```python
weighted_national = df.groupby("date").apply(
    lambda g: pd.Series({
        f"{col}_weighted": (g[col] * g[weight_col]).sum() / g[weight_col].sum()
        ...
    })
)
```

The analysis correctly shows the weighted average differs from the simple average (left-skewed distribution, KDE shifted). Despite this finding, the final model uses the simple average (`national_daily` from Cell 73). The effort invested in Cell 74–82 produces no downstream effect.

**Risk:** If the simple average introduces systematic bias (as the KDE in Cell 78 suggests), model accuracy for large, high-consumption states is degraded.

**Fix:** Either use `weighted_national` as the weather input, or document explicitly why simple averaging is acceptable (e.g., states are approximately equal in population-weighted energy share).

#### [LOW] `get_season()` uses a non-standard Indian meteorological definition
**Location:** Cell 26
```python
def get_season(month):
    if month in [12, 1, 2]: return 'Winter'
    elif month in [3, 4, 5]: return 'Summer'
    elif month in [6, 7, 8, 9]: return 'Monsoon'
    else: return 'Autumn'  # Oct, Nov
```
The Indian Meteorological Department defines pre-monsoon (March–May), monsoon (June–September), post-monsoon (October–November), and winter (December–February). The function matches this broadly, but October–November as "Autumn" is non-standard for Indian climate literature and may confuse domain readers.

**Fix:** Rename `'Autumn'` → `'Post-Monsoon'` to align with IMD terminology.

### 2. Edge Cases

#### [HIGH] Rolling windows with `min_periods=1` mask missing data
**Location:** Cells 31, 32
```python
merged["energymet_7d_avg"] = merged.groupby("states")["energy met (mu)"].transform(
    lambda x: x.rolling(window=7, min_periods=1).mean()
)
```
`min_periods=1` means the first 6 days of each state's series produce averages over 1–6 days, not 7. These early-window values silently enter the dataset and are used as features. For a state with sparse data this produces a systematically different feature distribution with no flag or warning.

**Fix:** Add a note or downstream check for how many rows have `< 7` days of rolling history. Alternatively, use `min_periods=7` and handle the resulting NaNs explicitly with `dropna()` before modelling.

---

#### [MEDIUM] 3-day centered rolling introduces future data leakage risk
**Location:** Cell 51
```python
merged['demand_rolling'] = merged.groupby('states')['energy met (mu)'].transform(
    lambda x: x.rolling(window=3, center=True).mean()
)
```
`center=True` means the rolling mean at day `t` includes data from day `t+1`. For retrospective EDA this is fine. However, `demand_rolling` is created on `merged` — the same DataFrame that feeds into the model — and the feature is later used as a predictor. If it leaks into training as-is, day `t`'s feature incorporates day `t+1`'s demand.

**Fix:** Confirm `demand_rolling` is excluded from `drop_features` in Cell 102, or replace `center=True` with `center=False` (trailing window) throughout.

### 3. Assumptions

#### [MEDIUM] Correlation interpretation conflates state-level and national-level analysis
**Location:** Cells 41–47, 87–94
The notebook correctly identifies that state-level demand vs. generation correlation (0.472) is weaker than national-level (≈1.0) and explains this well. However, the markdown at Cell 43 describes the 0.472 result as "surprising" without noting it is the expected outcome: state-level generation does not need to match state-level demand because the national grid transmits power between states.

This explanation only appears implicitly. A reader who skims may incorrectly believe the low correlation reflects poor data quality.

**Fix:** Add one explicit sentence: *"This is expected: India operates an interconnected national grid, so state-level generation need not match state-level demand."*

#### [MEDIUM] Zenodo dataset schema documented inline but not validated
**Location:** Cells 63–67
The dataset schema is described in a markdown cell (Cell 67) but no column validation is performed after loading:

```python
files = glob.glob('data/zenodo/*.csv')
for file in files:
    df = pd.read_csv(file)
    state_name = os.path.basename(file).replace(".csv", "")
    df['state'] = state_name.replace('_', ' ')
    dfs.append(df)
```

There is no check that every file has the expected columns, no dtype validation, and no missing-file detection. If a new state CSV is added without all columns, `pd.concat` will produce NaN-filled columns silently.

**Fix:**
```python
expected_cols = {'date', '2m_temperature_mean', 'utci_mean', ...}
for file in files:
    df = pd.read_csv(file)
    missing = expected_cols - set(df.columns)
    if missing:
        raise ValueError(f"{file} is missing columns: {missing}")
```

### 4. Complexity

#### [MEDIUM] Monthly resampling via `set_index` / `reset_index` is fragile
**Location:** Cells 34–35
```python
merged = merged.set_index('date')
merged.loc[total_mask, 'monthly_generation_avg'] = (
    merged[total_mask]
    .groupby(['states', merged[total_mask].index.to_period('M')])['generation_mu']
    .transform('mean')
)
merged = merged.reset_index(drop=False)
```
Setting the index on the full `merged` dataframe (which mixes `Total` and non-`Total` rows) to compute a `Total`-only column using a boolean mask and index manipulation is fragile. If the mask or index alignment shifts, non-`Total` rows get NaN silently.

**Simpler alternative:**
```python
monthly_avg = (
    merged[total_mask]
    .assign(period=merged.loc[total_mask, 'date'].dt.to_period('M'))
    .groupby(['states', 'period'])['generation_mu']
    .transform('mean')
)
merged.loc[total_mask, 'monthly_generation_avg'] = monthly_avg.values
```

### 5. Maintainability

#### [MEDIUM] State name normalisation is ad-hoc and distributed
**Location:** Cell 63, `app.py` (multiple locations)
In the notebook:
```python
state_name = os.path.basename(file).replace(".csv", "").replace('_', ' ')
```
In `app.py`, state names undergo a separate set of remappings ("J&K" ↔ "Jammu and Kashmir", "Delhi" ↔ "NCT of Delhi", etc.). These normalisation rules are scattered across both files and are not shared. A new state added to the Zenodo folder would require manual updates in both places.

**Fix:** Create a single `STATE_NAME_MAP` dictionary in a shared config or `utils/` module and import it in both `app.py` and the notebook.

## Positive Notes

- **Merge validation at Cell 12** — computing `match_rate` after the left join is excellent defensive practice and catches integration issues early.
- **Demand-weighted average (Cells 74–82)** — demonstrates strong domain awareness. The KDE and time-series overlay comparing simple vs. weighted averages are methodologically sound. The insight that population-weighted temperature is consistently higher is correct and important.
- **State-level vs. national-level correlation explanation (Cells 43–47, 94)** — the progressive investigation (observe anomaly → hypothesise → aggregate → confirm) is well-structured and teaches the reader about grid physics, not just model mechanics.
- **Using `holidays.India()` for holiday features** — using a maintained library rather than a hardcoded list is the right call; the library is updated annually.
- **`min_periods=1` with group-aware rolling (Cells 31–32)** — grouping by state before applying the rolling window correctly prevents bleed between state time series, which is a common mistake.
- **IQR outlier visualisation (Cell 22)** — even though no rows are dropped, computing per-column IQR and overlaying bounds on time series is better practice than a global zscore threshold.
- **30-day national demand smoothing (Cell 48)** — clipping outliers before plotting and overlaying raw vs. smoothed is clear and honest about data volatility.
- **Transparent schema documentation (Cell 67)** — the Zenodo dataset schema markdown is detailed and accurately reflects the Copernicus ERA5 variable naming convention.
