# Code Review — Electricity Demand Forecaster (Full Project)
**Scope:** Full codebase — `app.py`, `utils/`, `prompts/`, `model_metadata.json`, `requirements.txt`
**Reviewer:** Claude Code Review Agent
**Date:** 2026-03-13
**Rating: 7.5 / 10**

---

## Summary

This is a well-built, end-to-end electricity demand forecasting platform with a Streamlit UI, an XGBoost prediction engine, and a RAG-powered AI assistant. The ML pipeline is sound (proper time-series CV, log-transform with correct reversal, state-specific RMSE), the RAG architecture is clean and modular, and the locked-schema design in `rag_builder.py` / `prompts/assistant.md` is an excellent defence against LLM hallucination.

The main issues cluster around three areas:
1. **Brittle feature engineering** — hardcoded feature lists and duplicated state/region lookup tables that must stay in sync manually across multiple files.
2. **Silent data quality problems** — placeholder zeros for historical features that get fed to the model, undocumented `er` region remapping, and a `state_30d_baseline` hardcoded to `100.0` for states without historical data.
3. **Minor RAG correctness issues** — dimension mismatch risk on index reload, stale placeholder URL in HTTP headers, and a scope-checking list that will silently pass irrelevant queries containing common words.

### 1. Logic

#### [HIGH] Placeholder zeros for historical features fed directly to model
**Location:** [app.py:886–895](app.py#L886)

```python
df['generation_mu'] = 0.0        # Placeholder
df['energymet_7d_avg'] = 0.0     # Placeholder
df['generation_7d_avg'] = 0.0    # Placeholder
df['monthly_generation_avg'] = 0.0
df['monthly_energy_avg'] = 0.0
df['monthly_rolling_mean'] = 0.0
df['demand_rolling'] = 0.0
df['gen_rolling'] = 0.0
df['max.demand met during the day(mw)'] = 0.0
df['state_share'] = 0.0
```

All nine of these are in the model's `expected_features_base` list ([app.py:972–986](app.py#L972)) and will be passed to XGBoost as literal zeros. The model was trained with real values. Zeros are not out-of-distribution for all features (e.g. `generation_mu` = 0 implies a blackout), and the model has learned weights for them — so silently feeding zero during inference introduces a systematic bias that is invisible to the user.

**Fix:** Use `np.nan` or omit them from the feature matrix, then either retrain the model without these features or load climatological averages at inference time. At minimum, add a visible `st.warning()` that these features are set to zero and that forecast accuracy may be affected.

#### [HIGH] `state_30d_baseline` hardcoded to `100.0` as fallback
**Location:** [app.py:911](app.py#L911)

```python
df['state_30d_baseline'] = 100.0  # Default fallback
```

`state_30d_baseline` is used both as a model feature and as the denormalization multiplier. For states like Maharashtra (mean demand ~250 MU) or Manipur (~3 MU), `100.0` produces a wildly wrong prediction. There is no warning shown to the user.

**Fix:** Use the fallback baseline from `get_fallback_baseline(state_name)` here — the same function that is already used in `denormalize_predictions()` ([app.py:459](app.py#L459)) — to be consistent.

#### [HIGH] `er` region silently remapped to `nr`
**Location:** [app.py:1050–1051](app.py#L1050)

```python
if current_region == 'er':
    current_region = 'nr'  # Map to closest region
```

States Bihar, Chhattisgarh, Jharkhand, Odisha, and West Bengal all map to `er` (Eastern Region), which was present in training data but dropped from the model's one-hot features. They are silently remapped to `nr` (Northern Region) with no user-facing message. This means the model sees incorrect regional encoding for five large states.

The comment says "closest region" but Northern India is not geographically or climatologically close to Eastern India. West Bengal and Odisha have monsoon-dominated demand profiles unlike Rajasthan or Uttar Pradesh.

**Fix:** Either retrain including `er`, or at minimum display a `st.info()` message: *"Regional encoding for Eastern states (Bihar, Odisha, etc.) may reduce accuracy."*

#### [MEDIUM] `extreme_heat` threshold computed on the forecast window, not historical distribution
**Location:** [app.py:875–876](app.py#L875)

```python
heat_threshold = df['2m_temperature_mean'].quantile(0.95) if len(df) > 0 else 305.0
df['extreme_heat'] = (df['2m_temperature_mean'] > heat_threshold).astype(int)
```

For a 7-day forecast, the 95th percentile is computed over 7 rows. That means at most 1 day (ceiling of 5%) will ever be flagged, regardless of how hot all 7 days are. For a 30-day forecast, only 1–2 days get flagged. The flag is also used to drive high-risk day identification shown to users — so a heat wave week will mostly appear risk-free.

In training, the 95th percentile was computed over the full multi-year dataset. These are categorically different thresholds.

**Fix:** Use a fixed historical 95th-percentile threshold (e.g. `305.0 K` ≈ 31.85°C) hardcoded from the training distribution, not a per-forecast quantile.

#### [MEDIUM] Duplicate `all_states_ordered` list — second definition shadows the first
**Location:** [app.py:989–995](app.py#L989) and [app.py:1019–1025](app.py#L1019)

The same list is defined twice within `prepare_features_for_prediction()`. The first definition (lines 989–995) is used to build `expected_features` when `model.feature_names_in_` is unavailable, then immediately overwritten by the second definition (lines 1019–1025) before the loop. This is dead code and a maintenance trap — a developer editing one copy may not update the other.

**Fix:** Define once, reuse twice.

#### [MEDIUM] System prompt in `call_openrouter_llm` conflicts with `prompts/assistant.md`
**Location:** [utils/rag_engine.py:234–236](utils/rag_engine.py#L234)

```python
messages = [
    {"role": "system", "content": "You are a helpful AI assistant for electricity demand forecasting."},
    {"role": "user", "content": prompt}
]
```

The `system` message is a generic one-liner. The actual rules (locked schema, exact value reporting, no recomputation) are embedded in the `user` message via `prompt_template`. Some LLMs prioritise `system` over `user` instructions. The intent — strict rule-following — should be in the `system` role.

**Fix:** Move the full `prompts/assistant.md` content to the `system` message and put only `{user_query}` in the `user` message.

#### [LOW] `get_embedding_cached` ignores `use_local` parameter
**Location:** [utils/embeddings.py:142–156](utils/embeddings.py#L142)

`get_embedding_cached` calls `get_embeddings([text], model=model)` without passing `use_local`. Since `use_local=True` is the default, this is harmless today, but it means the cache key (`text`, `model`) does not encode which embedding source was actually used — if the API is tried first for some calls and local for others, returned vectors will have different dimensionalities (1536 vs 384) and be stored under the same cache key.

### 2. Edge Cases

#### [HIGH] `VectorStore.load_index()` can overwrite `self.dimension` with stale value
**Location:** [utils/vector_store.py:166](utils/vector_store.py#L166)

```python
self.dimension = data.get('dimension', self.dimension)
```

If the persisted index was built with 384-dimensional sentence-transformer embeddings and the code is later run with an API-based model producing 1536-dimensional embeddings, the loaded dimension will be 384. The FAISS index expects 384-dimensional queries. The next `add_documents` call will raise `ValueError: Embedding dimension mismatch` but the index is already loaded and in an inconsistent state.

**Fix:** Add a validation check after load:
```python
loaded_dim = data.get('dimension', self.dimension)
if loaded_dim != self.dimension:
    print(f"Warning: Dimension mismatch ({loaded_dim} vs {self.dimension}), rebuilding index")
    self.create_index()
else:
    self.dimension = loaded_dim
```

#### [HIGH] `denormalize_predictions` uses `pd.Timestamp.now()` to decide forecasting vs. historical mode
**Location:** [app.py:466–468](app.py#L466)

```python
first_date = pd.Timestamp(dates.iloc[0])
current_date = pd.Timestamp.now()

if first_date > current_date:
    # Forecasting future dates
```

This branch uses the 30-day tail of historical demand as the baseline. The `else` branch recomputes a per-date baseline. If a user runs a forecast for "today" or "yesterday" (e.g. to backtest), `first_date <= current_date` and the code takes the slower per-date path, computing `calculate_state_30d_baseline()` for each row. This is correct but inconsistent with the forecasting path, and the two paths use different baseline values for the same state on the same date.

Also, using wall-clock `now()` inside a cached or rerun Streamlit app means repeated runs at different times can produce different denormalizations for the same inputs.

**Fix:** Accept an explicit `is_forecast: bool` parameter rather than inferring from wall-clock time.

#### [MEDIUM] `save_index()` in `VectorStore` silently does nothing when `index_path` ends in a file with no parent dir
**Location:** [utils/vector_store.py:136](utils/vector_store.py#L136)

```python
os.makedirs(os.path.dirname(self.index_path), exist_ok=True)
```

If `index_path = "myindex.index"` (no directory component), `os.path.dirname()` returns `""`, and `os.makedirs("", exist_ok=True)` raises `FileNotFoundError` on some platforms. There is no try-catch around `save_index()`.

**Fix:** Guard with `if dir_name: os.makedirs(dir_name, exist_ok=True)`.

#### [MEDIUM] `generate_forecast_summary` iterates `iterrows()` to build date lists
**Location:** [app.py:1678](app.py#L1678)

```python
extreme_heat_dates = [row['date'].strftime('%Y-%m-%d') for _, row in extreme_heat_rows.iterrows()]
```

`iterrows()` is slow for large DataFrames. More importantly, `row['date']` may be a `str` rather than a `Timestamp` if the column type is not enforced, in which case `.strftime()` raises `AttributeError`.

**Fix:** Use vectorised operations:
```python
extreme_heat_dates = extreme_heat_rows['date'].dt.strftime('%Y-%m-%d').tolist()
```

#### [LOW] `is_query_in_scope` passes many off-topic queries
**Location:** [utils/rag_engine.py:69–77](utils/rag_engine.py#L69)

```python
in_scope_keywords = ['forecast', 'demand', 'electricity', ..., 'state', 'peak', 'average']
```

Words like `"state"`, `"average"`, `"error"`, `"model"`, and `"feature"` are extremely common in English. A query like *"What is the average population of Indian states?"* passes the scope check and reaches the LLM with energy-domain context injected. The keyword list is also hardcoded — no centralised config.

**Fix:** Require at least two domain keywords to match, or use bi-grams ("electricity demand", "energy forecast") for precision.

### 3. Assumptions

#### [HIGH] Model trained on 28 states; UI offers 36 — 8 states have no RMSE and use `100.0` baseline
**Location:** [app.py:95–104](app.py#L95), [model_metadata.json](model_metadata.json)

The UI dropdown includes Himachal Pradesh, Madhya Pradesh, Uttar Pradesh, NCT of Delhi, Dadra and Nagar Haveli, Daman and Diu, Lakshadweep, and Andaman and Nicobar — none of which appear in `model_metadata.json`'s `state_rmse`. For these states, `get_state_rmse()` returns `None`, confidence intervals are not shown, and `state_30d_baseline` falls back to `100.0`.

There is no user-facing warning that the prediction quality for these states is unknown.

**Fix:** Add a `st.warning()` for states not in `model_metadata.json`. Optionally grey them out in the dropdown with an "(unsupported)" suffix.

#### [MEDIUM] `state_to_region` defined twice in the same function with different contents
**Location:** [app.py:847–858](app.py#L847) and [app.py:1036–1047](app.py#L1047)

`engineer_features()` and `prepare_features_for_prediction()` each define their own `state_to_region` dict. The two copies are nearly identical but differ in coverage: the first includes `'Andaman and Nicobar': 'sr'`, the second does not. #### [MEDIUM] Open-Meteo `num_days` silently clamped to 16 with no UI feedback
**Location:** [app.py:527](app.py#L527)

```python
num_days = min(num_days, 16)
```

If a user selects a 30-day horizon, the API silently caps at 16 days and returns weather for only 16. The forecast still runs on 30 days by falling back to climatology for days 17–30, but the user sees no explanation of why the two data sources are mixed.

### 4. Complexity

#### [MEDIUM] `prepare_features_for_prediction` is 150+ lines with 4 hardcoded lists
**Location:** [app.py:934–1096](app.py#L934)

The function manually builds one-hot encoded columns for states, regions, seasons, and holidays using four separate hardcoded lists — all of which must exactly mirror what `scikit-learn`'s `get_dummies` produced during training. The function then constructs the feature matrix by iterating over `expected_features` and setting missing columns to 0.

When `model.feature_names_in_` is available ([app.py:1008–1009](app.py#L1008)), this entire manual process is bypassed. The fallback is therefore only used when the model object lacks `feature_names_in_`, which does not apply to XGBoost ≥ 1.6 — meaning the 100-line fallback path is likely never exercised in practice.

**Fix:** Remove the hardcoded fallback path. If `model.feature_names_in_` is unavailable, raise a clear error: *"Model does not expose feature names. Retrain with XGBoost ≥ 1.6 or save feature names separately."*

```python
df['day_of_week'] = df['date'].dt.dayofweek
df['dayofweek'] = df['date'].dt.dayofweek   # duplicate
```

### 5. Maintainability

#### [HIGH] Feature list in `expected_features_base` must be kept in sync with training manually
**Location:** [app.py:972–986](app.py#L972)

The feature list is described as coming from "the error message" (comment at line 971), meaning it was derived from a runtime XGBoost error and pasted in. When the model is retrained with any feature change, this list must be manually updated. There is no automated validation.

The `model.feature_names_in_` branch at line 1008 does read from the model object — but only as a fallback ordering mechanism. The upstream feature construction (lines 960–1058) still uses hardcoded lists to decide which columns to create.

**Fix:** Save `list(model.feature_names_in_)` to `model_metadata.json` at training time. Load it in `prepare_features_for_prediction()` and validate that all expected features are present before prediction.

`ALL_STATES` (36 states) and `all_states_ordered` (27 model-trained states) are different lists serving different purposes, but only one is defined at module level. **Locations:** [app.py:847](app.py#L847), [app.py:1036](app.py#L1036), and implicitly in the notebook

Three separate dict literals, two of which are inside the same 2,699-line file. If a state's grid region changes (e.g. Telangana moved from SR to a new region), all three must be updated.

#### [LOW] `@st.cache_data` on `load_model()` may mask model refresh
**Location:** [app.py:110](app.py#L110)

`@st.cache_data` serialises the model via pickle for caching. XGBoost `Booster` objects are not always pickle-safe across library versions. The correct decorator for resource objects (models, database connections) is `@st.cache_resource`, which avoids serialisation entirely.

### 6. Security

#### [MEDIUM] API key logged in error messages
**Location:** [utils/rag_engine.py:97–99](utils/rag_engine.py#L97)

```python
raise ValueError(
    "OPENROUTER_API_KEY not found in environment variables. "
    "Please set it in your .env file or environment."
)
```

This is safe — the key itself is not interpolated into the message. However, in [utils/embeddings.py:127](utils/embeddings.py#L127), the `str(e)` from an `HTTPError` may include the full request URL with headers, which can expose the API key in stack traces or Streamlit error displays.

**Fix:** Catch `requests.exceptions.HTTPError` and re-raise with a sanitised message that does not include the raw exception string.

#### [MEDIUM] `pickle.load` in `VectorStore.load_index()` loads arbitrary objects
**Location:** [utils/vector_store.py:163](utils/vector_store.py#L163)

```python
with open(data_path, 'rb') as f:
    data = pickle.load(f)
```

If `data/rag_index/` is writable by other processes or if the file is replaced by a malicious actor, `pickle.load` will execute arbitrary code. This is low risk in a local deployment but is worth noting if the app is ever deployed to a shared server.

**Fix:** For production, use `json` or `numpy.load` for the metadata payload; keep only the FAISS binary index in the `.index` file.

#### [LOW] No input validation on `state_name` before file path construction

## Positive Notes

- **`model.feature_names_in_` used as authoritative feature order** ([app.py:1008](app.py#L1008)) — using the XGBoost model's own stored feature names is the right approach; the hardcoded fallback should simply be removed.
- **Locked forecast schema in `rag_builder.py` + `prompts/assistant.md`** — the pattern of embedding a JSON snapshot of the forecast into the RAG document and explicitly forbidding LLM recomputation is elegant and correct.
- **`@st.cache_resource` on `initialize_rag_system()`** ([app.py:1708](app.py#L1708)) — correct decorator for a stateful object (VectorStore) that should not be serialised.
- **Cascading fallback in `get_embeddings()`** ([utils/embeddings.py:68–136](utils/embeddings.py#L68)) — local → API → local-on-failure is a robust resilience pattern.
- **L2 normalization before FAISS search** ([utils/vector_store.py:69](utils/vector_store.py#L69), [105](utils/vector_store.py#L105)) — normalizing both documents and queries before `IndexFlatL2` is the correct way to get cosine similarity semantics from a Euclidean index.
- **`TimeSeriesSplit` in model training** (notebook) — respects temporal ordering; no data leakage.
- **`holidays.India(years=range(2010, 2030))`** ([app.py:841](app.py#L841)) — pre-generating the holiday calendar across a wide range avoids repeated lookups and is efficient.
- **`get_fallback_baseline(state_name)`** — per-state fallback baselines rather than a single global default is the right design; just needs to be used consistently everywhere (see HIGH finding #2).
